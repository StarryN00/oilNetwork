# 企油通油站伙伴平台 V1 设计规格

日期：2026-06-25

## 1. 背景与定位

企油通油站伙伴平台 V1 面向民营油站，定位为“供给侧轻运营入口 + 增量效果看板”。它不是油站 ERP，也不替代油站现有收银、POS、零管或开票系统。

V1 的核心目标是跑通少量固定站点小闭环：油站提交站点、油品和优惠供给，企油通审核优惠并发布到现有司机端，订单从企油通系统同步回来，平台完成企业、司机、车辆、油站和优惠活动归因，最终让油站看到企油通带来的可识别交易、优惠效果和账单。

## 2. 已确认范围

V1 新做两端原型：

- 企油通内部运营后台 Web
- 油站端移动 Web / 小程序风格原型

不新做司机端。司机端继续由企油通现有司机端承接，展示经企油通审核发布的可用油站、油品和优惠，引导司机到站支付。

不做真实支付、银联/支付宝/微信接口、油站 POS 改造、电子发票接口、真实权限体系和生产数据库接入。原型采用模拟数据，但交互流程应按真实业务闭环设计。

## 3. 核心业务规则

### 3.1 油站端提交规则

油站端可提交和维护以下信息：

- 油站资料：站点名称、地址、联系人、电话、营业时间、开票能力、收款/结算备注。
- 油品信息：油品类型、挂牌价、可优惠价、适用车辆类型、更新时间。
- 优惠活动：优惠名称、适用油站、油品、优惠方式、优惠力度、适用企业/线路/司机/车辆范围、有效期、预算或限额。
- 对账反馈：订单异常反馈、月度账单确认。

油站资料和油品信息提交后直接生效，但必须记录提交人、提交时间和最近更新时间。企油通后台可纠错、下架或标记异常。

优惠活动必须经过企油通后台审核后才发布到司机端。油站端需要展示优惠状态：待审核、已发布、已驳回、已暂停、已结束。

### 3.2 企油通后台处理规则

企油通后台负责：

- 审核油站提交的优惠真实性和可执行性。
- 管理油站资料、油品、开票能力和合作状态。
- 将油站优惠匹配到物流企业、线路、司机或车辆。
- 将审核通过的优惠发布到企油通现有司机端。
- 从企油通系统同步订单、企业、司机、车辆、Token/额度、支付、发票等数据。
- 完成订单归因、异常处理、月度对账和效果报表。

### 3.3 增量归因口径

V1 采用保守口径：

默认增量 = 通过企油通司机端、Token、额度或活动入口，在该油站完成并可被企油通识别的交易。

驾驶舱和报表中拆分三类：

- 新客增量：该司机或车辆首次在此油站通过企油通交易。
- 复购增量：此前已通过企油通在该油站交易，后续继续交易。
- 活动增量：来自油站提交并经企油通审核发布的优惠活动。

V1 不要求证明车辆历史上绝对不会到站，只先证明交易可被企油通识别、运营和复盘。

## 4. 数据流

核心流向：

1. 油站端提交站点、油品、价格、优惠、开票能力和账单反馈。
2. 企油通中台审核优惠，匹配企业和线路，发布到司机端。
3. 司机在企油通现有司机端看到可用油站和优惠，到站完成支付。
4. 企油通系统同步交易订单、企业、司机、车辆、Token/额度、支付、优惠和发票状态。
5. 油站端和内部后台分别查看效果、异常和对账。

## 5. 页面结构

### 5.1 油站端移动端

#### 首页驾驶舱

目标：让油站快速看到企油通带来的可识别生意。

核心内容：

- 今日交易额、本月交易额
- 今日交易笔数、本月交易笔数
- 加油升数、客单价
- 新增司机数、复购司机数、复购率
- 已发放优惠
- 预计增量毛利
- 新客增量、复购增量、活动增量分布

#### 油站资料

目标：让油站维护基础供给信息。

核心内容：

- 站点名称、品牌、地址、定位
- 联系人、手机号
- 营业时间
- 支持油品
- 开票能力
- 合作状态
- 最近更新时间

提交后直接生效，后台可纠错或下架。

#### 油品与价格

目标：让油站维护可服务油品和价格。

核心内容：

- 油品类型
- 挂牌价
- 可优惠价
- 适用车辆类型
- 是否上架
- 最近更新时间

提交后直接生效，后台可纠错或下架。

#### 优惠活动

目标：让油站提交可控优惠，企油通审核后发布。

核心内容：

- 活动名称
- 活动油站
- 活动时间
- 活动油品
- 优惠方式：每升优惠、满减、固定金额立减
- 适用范围：全部企油通客户、指定企业、指定线路、指定司机、指定车辆
- 总预算、单车限额、单司机限额、每日限额
- 状态：待审核、已发布、已驳回、已暂停、已结束
- 活动效果：交易金额、交易笔数、优惠金额、参与司机数、复购情况

#### 订单与账单

目标：让油站查看订单、反馈异常、确认月账。

核心内容：

- 订单明细
- 优惠明细
- 发票状态
- 异常状态
- 月度账单
- 账单确认
- 异常反馈

### 5.2 企油通内部运营后台

#### 运营总览

目标：让内部团队掌握试点运行情况。

核心内容：

- 合作油站数
- 本月交易额、交易笔数、升数
- 活动交易额和优惠成本
- 待审核优惠
- 异常订单
- 待确认账单
- 重点试点站点

#### 油站管理

目标：管理油站供给资料和合作状态。

核心内容：

- 油站列表
- 油站详情
- 油品与价格
- 开票能力
- 合作状态
- 上下架控制
- 异常标记

#### 优惠审核与发布

目标：审核油站提交的优惠，并发布到司机端。

核心内容：

- 待审核优惠列表
- 优惠详情
- 适用企业、线路、司机、车辆配置
- 审核通过、驳回、暂停
- 发布状态
- 活动效果追踪

#### 客户与线路匹配

目标：把油站供给与物流企业需求连接起来。

核心内容：

- 物流企业列表
- 常用线路
- 常用站点
- 预估月度体量
- 推荐合作油站
- 固定站点试点标记

#### 订单归因与对账

目标：完成同步订单归因、异常处理和月度账单。

核心内容：

- 同步订单列表
- 企业、司机、车辆、Token/额度、活动归因
- 新客增量、复购增量、活动增量分类
- 异常订单处理
- 月度对账单
- 导出 CSV/Excel

## 6. 模拟数据字段

原型需要准备以下模拟数据对象：

- OilStation：油站资料、开票能力、合作状态
- FuelProduct：油品、挂牌价、可优惠价、上下架状态
- Campaign：优惠活动、审核状态、发布状态、预算和限额
- Enterprise：物流企业、线路、月度体量
- Driver：司机、手机号、所属企业
- Vehicle：车辆、车牌、所属企业和司机
- TokenOrQuota：Token/额度凭证、有效期、状态
- FuelOrder：订单、油品、升数、金额、优惠、支付状态、发票状态
- Reconciliation：月度账单、确认状态、导出状态
- AbnormalEvent：异常订单、异常原因、处理状态

## 7. 原型开发路径

采用方案 B：带模拟数据的交互原型。

要求：

- 做成真实前端应用，有路由、状态、表格、筛选、表单、审核流和模拟订单同步。
- 油站端和内部后台使用同一套模拟数据，体现提交、审核、发布、订单归因和账单确认闭环。
- 暂不接真实 API，但结构应方便后续替换为接口数据。
- UI 先服务演示和产品验证，不做生产级权限、审计、支付接口或复杂财务结算。

## 8. 成功标准

原型完成后应能演示以下闭环：

1. 油站在移动端维护油站资料和油品价格。
2. 油站提交优惠活动。
3. 企油通后台审核优惠并发布。
4. 模拟司机端承接发布后的优惠，不新做司机端完整页面。
5. 模拟订单同步到平台。
6. 系统将订单归因为新客增量、复购增量或活动增量。
7. 油站端看到驾驶舱和订单明细。
8. 企油通后台生成月度对账，油站端确认或反馈异常。

## 9. 暂不做内容

- 真实支付、云微、移企付、支付宝、微信支付接口
- 银联一键绑定 Token
- 油站 POS 或零管系统改造
- 真实电子发票接口
- 生产级账号权限和组织权限
- 复杂营销规则引擎
- 真实数据库和后端服务
- 独立司机端

## 10. 待后续实现计划确认

后续进入实现计划时，需要进一步确认技术栈、页面视觉风格、模拟数据规模、是否需要移动端独立路由，以及是否需要用本地存储保存演示过程中的表单修改。
